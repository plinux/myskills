# 性能测试

## 核心规则

- 当用户要求实际执行性能测试、benchmark、压测、perf 采样、火焰图或性能回归测量时使用本流程；不要在普通性能概念讨论、代码审查或不执行测试的分析中自动套用。
- 用户提供测试机器登录方式时，使用该机器测试；未提供时，默认在本地测试。
- 登录方式通常包括 SSH 主机、用户、端口、密钥、跳板机、工作目录和必要环境变量；缺少关键登录信息时先询问。
- 性能测试必须在同一测试机器上串行执行。不要同时运行多个 benchmark、压测、profiler、load generator 或其他会互相影响的测量任务。
- 不要用 subagent 或并行 shell lane 去并发执行性能测试。只允许并行做不会影响测试机器的只读准备工作。
- 缺少软件库、系统工具或需要调整系统配置时，必须先向用户确认，说明要安装或修改的具体内容。
- 用户明确要求“仅做性能测试”“只跑测试”“不要采集/分析”时，不启动额外采集，只按用户要求执行测试和记录结果。
- 用户明确指定结果保存路径、文件名、追加或覆盖方式时，按用户指定保存；不要使用本 Skill 的默认保存路径或命名规则覆盖用户要求。
- 默认性能测试要启动轻量观测：CPU breakdown、内存、IOPS/I/O、进程资源、短时 `perf` 或等价采样、短时堆栈采样。
- `sys%`、`nice%` 或类似比例异常偏高时，记录比例、时间段和对应负载，分析系统调用、调度、后台 nice 任务、内核配置或测试参数；疑似测试设置问题时补测确认。`iowait` 异常偏高表示 I/O 不足，必须先降低并发、降低 I/O 压力或调整数据/存储方式再重测。
- 用户要求“性能分析”时，先跑一次默认性能测试并包含轻量级采样，初步判断可疑瓶颈；随后可以使用完整 profiling、火焰图、trace、gdb、系统调用或锁分析等更重手段再跑一次测试来精确定位瓶颈。
- 重 profiling、gdb、trace 等可能明显影响性能结果的手段只能用于定位瓶颈，不要把它们的耗时当作基准性能结果。
- 性能测试完成后清理临时测试文件、临时采集文件和中间产物，避免长期占用空间；不要删除测试结果 Markdown 文档。
- 每次测试默认生成新的 Markdown 报告，不覆盖旧报告。只有用户明确要求追加或覆盖时才复用同一文件。

## 模式判断

- 仅测试模式：用户明确说只做测试、仅跑 benchmark、不要分析、不要采集、不要 profiling 时，只执行指定测试和必要结果记录。仍需做环境预检、串行锁和报告留档，除非用户明确指定更简化的输出。
- 默认测试模式：用户要求做性能测试但没有禁止采集时，执行 benchmark 并做低开销、短窗口采样。采样目标是解释结果可信度和初步瓶颈，不是覆盖整个测试周期。
- 用户指定保存模式：用户明确指定保存路径、文件名、报告格式、追加或覆盖策略时，该要求优先于默认 `performance-results/` 和默认文件命名。
- 性能分析模式：用户要求分析性能瓶颈、定位原因、找热点或解释性能差异时，先运行默认测试模式；基于初步证据选择第二轮更重的 profiling 或调试手段。第二轮采集应明确标记为诊断运行，与基准结果分开解释。

## 执行流程

1. 确认测试目标。

   如果用户提供远程登录方式，就在远程机器执行；否则在本地执行。记录主机名、OS/kernel、CPU 型号和核数、内存、磁盘、容器或虚拟机状态，以及实际工作目录。

2. 明确 benchmark 意图。

   从用户请求或仓库文档中提取工作负载、执行命令、数据集、测试时长、核心指标、通过标准、采集模式和结果保存方式。如果 benchmark 命令或目标指标无法安全推断，先问一个简短问题。

3. 获取执行锁。

   在测试机器上用原子方式加锁，例如 `mkdir /tmp/performance-testing.lock`。锁目录中记录 PID、时间戳、命令摘要和工作目录。若锁已存在，先检查现有任务，不要直接启动新的性能测试。测试结束后释放锁。

4. 检查环境资源。

   检查 CPU、内存、磁盘、网络、GPU 或其他设备资源是否满足本次测试要求。按需检查热降频、CPU governor、电源模式、后台负载、swap、磁盘空间和 I/O 压力。资源不足时停止测试并报告阻塞原因。

5. 检查软件依赖。

   测试前检查构建工具、运行时、benchmark 工具和观测工具是否可用。Linux 常见工具包括 `perf`、`pidstat`、`mpstat`、`iostat`、`vmstat`、`sar`、`top`、`ps`、`dmesg` 和 FlameGraph 脚本。macOS 上优先使用 `sample`、`spindump`、`powermetrics`、`vm_stat`、`top` 等可用替代工具。缺少必要软件时，先请求用户确认是否自动安装。

6. 建立基线。

   在正式测试前采集空闲 CPU、内存、磁盘和进程状态。对于高风险、长时间或从代码中推断出的测试，先跑短 smoke test。仅测试模式下只保留必要基线信息，不做额外 profiler 采集。

7. 串行执行测试。

   按顺序执行 warmup 和 measured iterations。可比较的多轮测试之间保持输入、commit、环境、系统配置和数据集稳定。记录命令行、耗时、退出码和原始 stdout/stderr 路径。性能分析模式下，第一轮是基准测试，第二轮诊断测试必须单独标记。

8. 短时采样观测数据。

   默认测试模式下，在代表性 steady-state 阶段采集低开销短样本。CPU、内存、IOPS/I/O 和进程资源采样可用保守间隔覆盖测试窗口；CPU 采样必须能区分 user/system/nice/iowait/idle 等比例。`perf record`、堆栈采样、火焰图原始采集和 trace 建议 10-60 秒，未经用户确认不要超过 120 秒。仅测试模式跳过这些额外采集。性能分析模式可在第二轮使用更完整 profiling、gdb 或 trace，但要明确其诊断性质和性能扰动。

9. 分析结果。

   汇总吞吐、延迟、方差、资源饱和、热点函数、堆栈证据和异常现象。区分实测证据和推断结论。CPU `sys%`、`nice%` 异常偏高时必须解释可能原因；若怀疑测试设置导致，调整设置后补测并用补测结果确认。`iowait` 异常偏高时判定为 I/O 压力或能力瓶颈，必须降低并发或 I/O 负载后补测，不要把原结果当作目标并发下的纯 CPU 性能。明确指出噪声源，例如后台任务、热降频、swap、缓存效应、dirty git state 或虚拟化影响。

10. 写入报告。

   用户指定保存方式时按用户指定执行。未指定时，每次生成唯一 Markdown 文件。如果目标目录是 git 仓库，记录当前 commit id、分支和 dirty state。文件名使用 `YYYYMMDD-HHMMSS-<shortcommit>-performance-test.md`；没有 commit 时使用 `YYYYMMDD-HHMMSS-performance-test.md`。在 git 仓库中优先保存到 `performance-results/`；非仓库则保存到当前工作目录。

11. 清理临时文件。

   测试结束后删除临时测试输入、scratch 目录、临时日志、未被报告引用的大型中间采集文件和中间转换产物。保留 Markdown 结果文档。若需要保留较大的原始证据文件，必须在报告中列出路径、大小和保留原因；否则将关键信息摘要写入 Markdown 后清理原始临时文件。

## 报告内容

Markdown 报告必须包含：

- 摘要：测试内容、测试机器、时间和核心结论。
- 测试目标：本地或远程登录方式、主机信息、OS/kernel、CPU、内存、磁盘、电源或容器背景。
- Git 状态：仓库路径、commit id、分支、dirty state；非 git 目录要明确说明。
- 环境预检：资源检查、依赖检查、缺失项、用户确认的安装或跳过项。
- 执行命令：构建、warmup、benchmark 和观测采集命令，包含工作目录。
- 测试结果：指标、迭代表、方差、退出状态和原始证据路径。
- 观测信息：采集模式、CPU breakdown、内存、IOPS/I/O、`perf`、火焰图、堆栈和资源采样窗口，包含采样时长和开销说明；仅测试模式要明确说明用户要求不采集。
- 分析判断：瓶颈、证据支持的解释、性能变化和置信度。
- 异常补测：记录 `sys%`、`nice%`、`iowait` 等异常比例、原因判断、采取的降并发或降 I/O 负载措施、补测命令和补测结果；没有补测时说明阻塞原因。
- 性能分析追加项：第一轮基准结果、初步瓶颈判断、第二轮诊断手段、诊断结果，以及哪些数据不能作为基准性能数字。
- 清理记录：删除了哪些临时文件，保留了哪些结果文档或必要证据文件。
- 限制：未控制变量、失败的采集器、缺少工具和后续提升置信度的方法。

## 安全边界

- 先做低侵入测量，再根据基线结果追加更重的 profiling；仅测试模式不追加采集。
- 报告保存方式以用户明确要求为准；用户未指定时才使用默认目录和命名。
- 报告和需要保留的证据文件要放在同一结果目录，或在报告中用相对路径链接。
- 默认清理临时测试文件和中间采集文件；不要删除测试结果 Markdown 文档。
- 不要覆盖旧报告，除非用户明确要求追加或覆盖。
