# Code Logic Optimizer

用于把最近代码变更中的可维护性问题转成可验证、可提交的逻辑优化。默认保持行为不变，优先优化结构、重复、分支、命名、错误处理和边界清晰度。

## 目标范围

1. 若用户指定文件、提交或范围，按用户范围执行。
2. 否则先检查未提交变更；存在未提交变更时，只优化这些变更涉及的代码。
3. 若工作区干净，则优化最后一个提交引入或修改的代码。
4. 先读取项目 `AGENTS.md`、`CLAUDE.md`、README、测试说明和相关代码风格；更近的项目规则优先。
5. 不要把无关文件、用户未要求的重构或格式化混入优化项。

## 历史记录

每轮审查前先查找当前范围相关的既有优化计划、审查报告和修复记录，只读取文件名或内容命中 `code-logic`、`optimization`、`code-review`、`simplifier`、`refactor` 的记录。默认检索：

- `docs/code-logic-optimization/`
- `docs/code-review/`
- `.codex/`
- `.claude/`
- 用户指定的记录路径

若项目规则声明这些目录为本地文件，不要提交它们。找不到历史记录时，在本轮计划文件中写明“未发现相关历史记录”。

## 审查与简化

1. 尽可能并行运行独立审查通道：
   - `oh-my-codex:code-review`、`code-review` 或其他可用 Code Review Skill。
   - `code-simplifier` 或其他代码简化、提炼、去复杂度 Skill。
2. 优先使用子智能体并行执行只读审查；工具不可用时，由当前智能体按同等标准执行，并在计划中标记不可用通道。
3. 汇总结果时去重并分类：
   - `safe`: 行为保持不变、可直接实施。
   - `needs-decision`: 涉及 API、性能/正确性权衡、兼容性或业务语义。
   - `reject`: 过度抽象、偏离用户范围、无法证明收益或会改变行为。
4. 如果不同轮次或不同通道对同一问题给出相反建议，不要自动来回修改；把冲突写入计划并让用户判断。
5. 实施前检查历史记录，避免重新引入已经被审查指出的问题。

## 计划文件

除非用户明确要求只给建议或先交互确认，否则直接创建可实施计划文件，再按计划执行。默认路径：

```text
docs/code-logic-optimization/YYYYMMDD-HHMMSS-plan.md
```

计划结构使用 `references/optimization-plan-template.md`。若项目规则禁止提交 `docs/`，计划文件只作为本地工作记录，不放进优化提交。

## 实施规则

1. 只实施 `safe` 项；`needs-decision` 项保留给用户确认。
2. 每个优化项独立修改、独立验证、独立提交。
3. 保持外部行为不变：不要改变公共 API、错误语义、事务边界、并发顺序、权限、安全边界、数据格式或性能测试口径。
4. 优化过程中发现计划不准确时，先更新计划文件，再继续实施。
5. 每个提交只包含一个优化项及其必要测试；提交信息遵守目标仓库规则。
6. 不要提交历史审查记录、计划文件或临时材料，除非用户或目标仓库明确要求。

## 循环复审

完成一轮优化后重新执行“历史记录 -> 审查与简化 -> 汇总”流程：

- 发现新的 `safe` 项：追加到计划并继续逐项实施。
- 只发现重复项：记录为已处理，不再修改。
- 发现冲突建议：记录冲突并停止对应项，等待用户判断。
- 没有新的安全优化项：停止。

## 验证与输出

每个优化项至少运行相关测试或静态检查；没有可运行测试时，记录原因并做代码级 diff 自查。最终用中文汇报：

- 实际优化范围。
- 计划文件路径。
- 创建的提交列表。
- 每个提交的验证结果。
- 跳过、拒绝或需要用户决策的项目。
