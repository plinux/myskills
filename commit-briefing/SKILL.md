# Commit Briefing

生成仓库提交历史简报。默认分析整个仓库历史；用户指定提交、分支、tag 或 range 时，以用户范围为准。

## 范围

1. 未指定范围时，分析当前仓库所有可达提交。
2. 指定范围时，按用户给出的表达式执行，例如 `main..HEAD`、`v1.0..v1.1`、`HEAD~20..HEAD` 或单个 commit。
3. 简报只基于 Git 历史和可读取证据；不要补写没有证据的测试结论。

## 数据采集

优先运行脚本获取基础数据：

```bash
python3 <skill-dir>/scripts/commit_brief.py --repo <repo> --format json
python3 <skill-dir>/scripts/commit_brief.py --repo <repo> --range main..HEAD --format markdown
```

脚本统计：

- commit 分类：从标题前缀、conventional commit 类型和变更文件推断。
- 代码量：非测试代码文件的新增/删除行。
- 测试代码量：测试目录、测试文件和常见测试后缀的新增/删除行。
- 其他资源：图片、图标、字体、音视频、压缩包、二进制资源等。
- 文档、配置和其他文件变更。
- 测试验收证据：只从 commit message/body 中显式的测试、验证、覆盖率、通过/失败记录提取。

## 测试证据规则

- 只有存在明确证据时才显示测试验收情况。
- 可以作为证据的内容包括：`Tested:`、`Tests:`、`Verification:`、`测试:`、`验证:`、测试命令输出、CI 结果摘要、验收记录、覆盖率数字。
- 只有看到通过/失败词或 CI 状态时才写“是否通过”。
- 只有看到覆盖率数字时才写覆盖率。
- 不要因为 commit 修改了测试文件、CI 配置或文档中提到测试，就推断该 commit 已完成测试验收。

## 输出格式

默认用中文输出，按提交时间从旧到新排列。每个 commit 至少包含：

- commit 短 hash 和标题。
- 分类。
- 代码量：新增/删除行和文件数。
- 测试代码量：新增/删除行和文件数。
- 资源变更：资源文件数和主要类型。

仅在有证据时追加：

- 测试/验收命令或记录。
- 是否通过。
- 覆盖率。

最后给出范围汇总：提交数、各分类数量、代码总量、测试代码总量、资源文件数，以及没有测试证据的提交数量。

## 质量要求

- 分类不确定时写“其他”，不要强行归类。
- 二进制文件没有行数时只统计文件数和类型。
- merge commit 若没有可读 diff，仍列出标题和分类，并标记代码量为 0。
- 输出不要包含推测性测试结论。
- 用户要求保存文件时，再把简报写入用户指定路径；否则直接在回复中给出。
