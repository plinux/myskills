# Digsight DXDCNet Protocol

用于处理动芯 Digsight DXDCNet 通讯协议相关任务，包括阅读抓包、编写编解码器、补齐协议常量、生成测试向量、解释设备状态、排查 D9000/Booster/Detector/附件控制/机车控制通信问题。

## 触发场景

- 用户提到 `DXDCNet`、`DxdcNet`、`Digsight`、动芯、D9000、6103、RailCom、DCC 控制器、Booster、Detector、附件解码器、道岔解码器或协议 V3.11。
- 需要解析或构造 DXDCNet 的 UDP 帧、CAN 标准帧、CAN 扩展帧、网络 ID 申请、MAC 地址广播、机车速度/功能控制、附件/道岔控制、设备状态或参数读写。
- 需要把协议字段转成代码常量、枚举、结构体、测试样例、抓包解释或调试报告。

## 使用流程

1. 先确认传输层：上位机通讯是 UDP；模块间通讯是 CAN 250000 b/s。
2. 确认目标是解析、构造、实现、测试还是调试。实现时保留原始字节、字段名和位图，不要只输出解释后的字符串。
3. 读取 `references/protocol.md` 中相关小节：
   - 帧头、设备类型、命令字：读“基础结构”和“命令字速查”。
   - 单条命令载荷：读“载荷定义”中对应编号。
   - UDP 抓包：读“UDP 封装”和“已确认 UDP 证据”。
   - D9000 参数：读“参数地址”。
4. 遇到协议证据冲突时，以实机抓包和真实设备回包为最高优先级；PDF、网页文档、旧 Demo 或反编译 App 只能作为辅助证据。在结果中标注冲突并保留原始包。
5. 生成代码时，优先建立常量和小型 encode/decode 函数；为每个已实现命令增加至少一个字节级测试。
6. 调试抓包时，按顺序检查包头、长度、源设备类型、源 ID、命令字、DLC/数据长度、载荷字段和校验/应答关系。

## 关键约束

- PDF 未给出 UDP 校验算法，但官方 `DigsightDemo_2025_12` C# 源码和 2026-06-20 实机包已确认 Demo UDP 帧使用从 `FF FF` 包头开始到校验前一字节的 XOR；实现时仍保留原始帧并标注证据来源。
- 厂家已说明文档可能不准确；当文档、旧 Demo、App 反编译和真实抓包互相冲突时，优先采用抓包/真实回包，并把其它来源作为冲突备注。
- 官方 Android App 默认目标 UDP 端口为 `12000`，本地端口为 `6667`，ThrottleID 默认为 `1`；官方 PC Demo 默认本地端口为 `12001`。直连设备优先按目标 `12000` 测试。
- 官方 Android App V3 已确认 CV 编程使用 `0x14` Programmer direct read/write，编程轨 op=`0`，主轨机车 POM op=`6` 并附加两字节 POM 车辆地址；回包为 `0x17` value 或 `0x15` ACK；不要用 `0x8A` 原始 DCC packet 封装替代 CV 编程路径。
- 官方 Android App 反解析确认机车速度/功能控制会先发 `0x04` 请求机车控制，但未收到 `0x07` 控制权 ACK 时仍允许继续发送后续 `0x10/0x11` 用户操作；收到明确拒绝 ACK 时才应阻断。该规则是动芯 App 兼容行为，不要推广为通用 DCC 或其它控制器规则。
- 官方 Android App V3 的 N/HO/G 输出枚举为 N `0x78`、HO `0xA0`、G `0xB4`，与 PDF/旧 Demo 中 N `0xE8`、HO `0xF3`、G `0xF9` 存在版本冲突；实现当前 D9000 App 兼容路径时优先使用 V3 App 值，并在文档中标注冲突。
- 2026-06-20 用户 App 设置与实机只读回包确认 D9000 `0x81/0x82/0x83/0x84` 分别为 N/HO/G/DC 限流参数，原始值按 `40 mA` 倍数换算；旧文档或旧记录中的 `20 mA` 换算已被实测推翻。
- 2026-06-20 实测确认 D9000 参数 `0x7E` 为屏幕亮度，参数 `0x80` 为屏幕方向；原始值 `0x02` 对应用户 App 设置后的“右”，方向按顺时针递增循环，映射为 `0x02=右`、`0x03=下`、`0x00=左`、`0x01=上`。
- UDP 小节写作 `0CCCCCCC`，但命令表包含 `0x80`、`0xE0` 等高位为 1 的命令；实现时不要无依据地屏蔽命令字最高位。
- CAN 表格中的 `DLC` 包含命令字本身；例如 `DLC=5` 通常表示 1 字节命令加 4 字节载荷。
- 机车地址字段按表格使用 `Low`、`High` 两字节；不要在未验证前改成网络字节序。
- 设备类型和网络 ID 是两个概念：设备类型是 4 bit，网络 ID/器件地址通常是 7 bit。
- 扩展帧中的网络 ID 申请和 MAC 地址字段有专用格式；不要套用普通标准帧命令格式。
