# Bug Root Cause Analysis

用于分析实测中发现的崩溃和 bug 根因。核心原则：先保护现场和采集证据，再提出假设；根因必须由复现用例、现场日志、堆栈或等价证据支撑。

## 现场采集

1. 先确认用户描述的问题：现象、触发步骤、发生时间、影响范围、频率、最近变更和期望行为。
2. 优先检查是否仍有环境现场：进程状态、core dump、crash dump、日志目录、配置、输入数据、版本号、运行参数、依赖版本、系统资源和外部服务状态。
3. 程序自身日志优先于系统推测；同时收集 stderr/stdout、应用日志、测试日志、系统日志、监控指标和调用链证据。
4. 对真实环境先做只读采集；不要清理日志、重启服务、删除临时文件或改变数据，除非用户明确授权。
5. 没有现场时，向用户列出缺失证据和最小补充材料，不要直接断言根因。

## 初步分析

1. 按证据列出候选原因，并标注证据强度、反证和待验证点。
2. 把环境因素和代码因素分开：版本/配置/资源/输入/并发/外部依赖属于环境侧；异常路径、边界条件、状态机、并发控制和错误处理属于代码侧。
3. 先解释为什么某个假设能覆盖已知现象，再设计验证方法。
4. 不要把“看起来像”写成根因；无法证明时写“候选原因”。

## 复现确认

1. 优先编写最小复现用例、回归测试或可重复脚本，尽量固定输入、配置、随机种子、并发度和环境变量。
2. 复现用例必须先能稳定暴露原问题；若用例无法失败，不能用它证明根因。
3. 能用复现用例重现崩溃或 bug，并能用现场证据解释触发路径时，才认为根因基本确定。
4. 根据复现现场继续分析堆栈、日志、变量状态和代码路径；必要时增加临时诊断输出辅助定位。
5. 找到根因后，给出修改方案、风险、验证命令和建议补充的回归测试。

## 调试日志授权

只有在现有现场和复现尝试不足以确认根因时，才建议增加调试日志或诊断探针。

- 修改代码增加调试日志前，必须先向用户申请授权。
- 用户授权后，日志必须使用明显标记，例如 `RCA_DEBUG`、`TEMP_RCA_LOG` 或项目认可的临时标识。
- 记录每个调试日志的位置、目的、预期观察值和后续还原方式。
- 调试日志只用于定位根因；找到问题后必须提醒用户移除或收敛为正式日志。
- 未获授权时，不要修改代码加入调试日志。

## 实验限制

- 默认最多尝试 10 个候选原因或复现实验。
- 每次实验都记录：假设、操作、输入、观察结果、是否支持假设、下一步。
- 10 次仍无法确定根因时，暂停并询问用户：继续尝试其他原因，还是先补充现场/授权增加调试日志。
- 如果用户明确要求“一直找到原因”为止，则持续迭代，但仍要记录每轮证据和排除项。

## 输出要求

最终回复默认用中文，包含：

- 问题现象和复现范围。
- 已采集证据：环境、日志、堆栈、配置、输入和监控。
- 候选原因与排除过程。
- 复现用例或无法复现的说明。
- 根因结论和证据链；没有证据时写“不确定”，不要硬下结论。
- 推荐修改方案、验证命令、回归测试建议。
- 若加过调试日志，列出所有临时日志标记和还原位置。
